Create `Dockerfile` at repo root using multi-stage build with uv workspace pattern from the research:

**Stage 1 (builder):**
- Base: `public.ecr.aws/lambda/python:3.14` (x86_64, matching current Terraform which does NOT set architectures)
- Copy uv binary from `ghcr.io/astral-sh/uv:0.10`
- Set `UV_COMPILE_BYTECODE=1` and `UV_NO_INSTALLER_METADATA=1` for cold start optimization
- WORKDIR `/build`
- Copy workspace root files: `pyproject.toml`, `uv.lock`
- Copy workspace member pyproject.toml files: `utils/pyproject.toml`, `backend/pyproject.toml`
- First RUN: `uv export --frozen --no-emit-workspace --no-dev --package ferry-backend -o requirements.txt` then `uv pip install --no-cache-dir -r requirements.txt --target /build/deps` (third-party deps only, cached layer)
- Copy workspace source: `utils/src`, `backend/src`
- Second RUN: `uv export --frozen --no-dev --no-editable --package ferry-backend -o requirements-all.txt` then `uv pip install --no-cache-dir -r requirements-all.txt --target /build/all` (everything including workspace members)

**Stage 2 (runtime):**
- Base: `public.ecr.aws/lambda/python:3.14`
- `COPY --from=builder /build/all ${LAMBDA_TASK_ROOT}`
- CMD: `["ferry_backend.webhook.handler.handler"]`

Key: Use `--mount=type=cache,target=/root/.cache/uv` on both RUN steps. Do NOT use `COPY . .` -- copy only what's needed. The first RUN (third-party deps) changes only when dependencies change; the second RUN changes when source changes. This gives optimal layer caching.

Modify `backend/src/ferry_backend/settings.py`:

1. Add `import boto3` at top (already a dependency of ferry-backend)
2. Add `from pydantic import model_validator` to existing pydantic import
3. Add three new optional fields with empty string defaults:
   - `app_id_secret: str = ""`  (maps to `FERRY_APP_ID_SECRET` env var)
   - `private_key_secret: str = ""`  (maps to `FERRY_PRIVATE_KEY_SECRET`)
   - `webhook_secret_secret: str = ""`  (maps to `FERRY_WEBHOOK_SECRET_SECRET`)
4. Change `app_id`, `private_key`, `webhook_secret` to have default `""` (they'll be populated either from env vars or SM resolution)
5. Add `@model_validator(mode="after")` method `resolve_secrets`:
   - Define mapping: `{"app_id_secret": "app_id", "private_key_secret": "private_key", "webhook_secret_secret": "webhook_secret"}`
   - For each mapping pair, check if the source field (e.g., `app_id_secret`) has a non-empty value
   - If any secrets to resolve: create `boto3.client("secretsmanager")` and call `get_secret_value(SecretId=secret_name)` for each
   - Use `object.__setattr__(self, field, resp["SecretString"])` to set the resolved value (bypasses Pydantic frozen model protection)
   - Return `self`
6. Keep the existing `strip_private_key_whitespace` field_validator -- it runs BEFORE the model_validator

Per user decision: local dev sets `FERRY_APP_ID`, `FERRY_PRIVATE_KEY`, `FERRY_WEBHOOK_SECRET` directly. Lambda sets `FERRY_APP_ID_SECRET=ferry/github-app/app-id`, etc. The _SECRET suffix env vars come from Terraform (already set in `lambdas.tf` as `FERRY_APP_ID_SECRET`, `FERRY_PRIVATE_KEY_SECRET`, `FERRY_WEBHOOK_SECRET_SECRET`).

The double "SECRET" in `FERRY_WEBHOOK_SECRET_SECRET` is intentional -- first "SECRET" is part of the field name (`webhook_secret`), second is the `_SECRET` suffix pattern.
