## Summary

This phase performs a pure directory restructure of the Ferry IaC layout with corresponding S3 state key migration. There are 4 Terraform projects to move, each requiring a directory move (git), a backend key update in `providers.tf`, and a `terraform init -migrate-state` to copy state to the new S3 key. The shared project also has a `terraform_remote_state` reference to the OIDC project that must be updated to the new key.

The migration is straightforward but operationally sensitive -- a mistake in state key handling could orphan remote state or cause Terraform to believe resources need recreation. The approach is well-supported by Terraform's built-in `-migrate-state` flag, which copies state from the old key to the new key in S3 (same bucket). The old keys are NOT automatically deleted, which provides a natural backup.

**Primary recommendation:** Write a sequential shell script that for each project: (1) moves directory via git, (2) updates providers.tf backend key, (3) updates any remote_state references, (4) deletes `.terraform/` directory, (5) runs `terraform init -migrate-state -force-copy -input=false`, (6) validates with `terraform plan -input=false`. Fail fast on any error. Order: backend, ecr, oidc, shared.

### Pattern: Per-Project Migration Sequence
**What:** Each of the 4 TF projects follows an identical migration sequence.
**When to use:** Every project in the migration.
**Steps:**
1. `git mv` source directory to target directory
2. Update `key` in `backend "s3"` block in `providers.tf`
3. Update any `terraform_remote_state` data source `key` values (only shared/data.tf has one)
4. Delete `.terraform/` directory (clean slate -- avoids stale provider/module cache pointing to old paths)
5. `terraform init -migrate-state -force-copy -input=false` from new directory
6. `terraform plan -input=false` to verify "No changes"

### Pattern: Migration Script Structure
**What:** A single shell script that runs all 4 migrations sequentially with fail-fast.
**Recommended structure:**
```bash
#!/usr/bin/env bash
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"

# Helper function for each migration
migrate_project() {
    local src="$1" dst="$2" name="$3"
    echo "=== Migrating: $name ==="

    # Step 1: git mv
    git mv "$REPO_ROOT/$src" "$REPO_ROOT/$dst"

    # Step 2-3: providers.tf and data.tf updates already done in code

    # Step 4: Clean .terraform/
    rm -rf "$REPO_ROOT/$dst/.terraform"

    # Step 5: Reinitialize with state migration
    terraform -chdir="$REPO_ROOT/$dst" init -migrate-state -force-copy -input=false

    # Step 6: Validate
    terraform -chdir="$REPO_ROOT/$dst" plan -input=false

    echo "=== $name: OK ==="
}
```

### Anti-Patterns to Avoid
- **Moving directories before updating providers.tf:** The `git mv` moves files, but providers.tf must have the NEW key before `terraform init` is run. Both must happen together -- update the key in code first (or as part of the same step), then init.
- **Running `terraform init` without deleting `.terraform/`:** The old `.terraform/` contains cached provider plugins and a `terraform.tfstate` that records the OLD backend config. Leaving it in place can cause init to use stale backend config or skip migration.
- **Migrating shared before OIDC:** Shared references OIDC via `terraform_remote_state`. If OIDC hasn't been migrated yet, the remote state reference will break. OIDC must migrate first.
- **Using `terraform state mv`:** This command moves resources within a state file. It does NOT move state files between S3 keys. The correct tool is `terraform init -migrate-state`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| State file copying between S3 keys | `aws s3 cp` of .tfstate files | `terraform init -migrate-state -force-copy` | Terraform's built-in migration handles state format validation, locking, and backend metadata tracking in `.terraform/terraform.tfstate` |
| Non-interactive prompts | Complex expect scripts | `-force-copy -input=false` flags | Terraform natively supports non-interactive migration |
| Directory moves with git history | `cp` + `rm` | `git mv` | Preserves git blame/history tracking |

**Key insight:** Terraform's `-migrate-state` with `-force-copy` handles the entire state transfer workflow. The migration script is really just orchestration around this built-in capability.

## Common Pitfalls

### Pitfall 1: Stale .terraform/terraform.tfstate
**What goes wrong:** After moving directories, the `.terraform/terraform.tfstate` file in the old location records the old backend config. If not deleted, `terraform init` may use it as the source of truth and not detect the key change.
**Why it happens:** `.terraform/terraform.tfstate` is Terraform's local record of which backend is currently configured. It's NOT the actual state -- it's metadata about where state lives.
**How to avoid:** Delete the entire `.terraform/` directory before running `init -migrate-state` in the new location. The CONTEXT.md explicitly requires this.
**Warning signs:** `terraform init` succeeds without prompting for migration.

### Pitfall 2: Remote State Reference Not Updated
**What goes wrong:** After migrating OIDC to its new key, if `data.tf` in the shared project still references the old key (`global/aws/oidc/terraform.tfstate`), `terraform plan` will fail to read remote state.
**Why it happens:** `terraform_remote_state` data sources have hardcoded S3 keys. Moving the state file without updating references breaks the chain.
**How to avoid:** Update the `key` in the `terraform_remote_state "oidc"` block in shared's `data.tf` at the same time as the directory move.
**Warning signs:** `terraform plan` errors with "Failed to load state" or similar remote state read failure.

### Pitfall 3: git mv Parent Directory Creation
**What goes wrong:** `git mv iac/global/aws/backend iac/global/cloud/aws/backend` fails because `iac/global/cloud/aws/` doesn't exist yet.
**Why it happens:** `git mv` does not create intermediate directories.
**How to avoid:** Create target parent directories with `mkdir -p` before running `git mv`.
**Warning signs:** `git mv` exits with error "destination directory does not exist".

### Pitfall 4: Ordering -- Backend Must Be First
**What goes wrong:** If the backend project (which manages the S3 bucket itself) is migrated after other projects, you could end up in a state where other projects' state is migrated but the backend project's own state is still at the old key.
**Why it happens:** All projects use the same S3 bucket. Backend manages the bucket, but its own state is just another key in that bucket.
**How to avoid:** Migrate backend FIRST. It's self-contained and doesn't reference other projects' remote state.
**Warning signs:** No immediate error, but conceptual dependency ordering is wrong.

### Pitfall 5: .terraform.lock.hcl Handling
**What goes wrong:** Lock files exist on disk but are in `.gitignore`. After `git mv`, they won't be moved. After deleting `.terraform/`, running `init` will regenerate lock files.
**Why it happens:** `.terraform.lock.hcl` is gitignored in this project. `git mv` won't move untracked files.
**How to avoid:** Since lock files are gitignored, they just need to be regenerated by `terraform init`. No special handling needed -- deleting `.terraform/` and running `init` handles everything.
**Warning signs:** None -- this is a non-issue given the project's .gitignore config.

### Pitfall 6: Old S3 Keys Left Behind
**What goes wrong:** `terraform init -migrate-state` COPIES state to the new key but does NOT delete the old key from S3. Old state files remain at the old keys indefinitely.
**Why it happens:** Terraform copies, it doesn't move. This is by design for safety.
**How to avoid:** This is actually a feature -- old keys serve as natural backups. Optionally clean up old keys after validating all migrations succeed. The S3 bucket has versioning enabled, providing additional safety.
**Warning signs:** None -- this is expected behavior.

### Migration Script Skeleton
```bash
#!/usr/bin/env bash
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
IAC="$REPO_ROOT/iac"

echo "Ferry IaC Directory Restructure and State Migration"
echo "===================================================="
echo ""

# Pre-flight: verify terraform is available
command -v terraform >/dev/null 2>&1 || { echo "ERROR: terraform not found"; exit 1; }

# Pre-flight: verify we're in the repo
[[ -d "$IAC/global/aws/backend" ]] || { echo "ERROR: Expected directory not found: $IAC/global/aws/backend"; exit 1; }

#----------------------------------------------------------------------
# Step 1: Create target directories
#----------------------------------------------------------------------
mkdir -p "$IAC/global/cloud/aws"
mkdir -p "$IAC/aws/staging"

#----------------------------------------------------------------------
# Step 2: Migrate global/backend
#----------------------------------------------------------------------
echo "[1/4] Migrating backend..."
rm -rf "$IAC/global/aws/backend/.terraform"
git mv "$IAC/global/aws/backend" "$IAC/global/cloud/aws/backend"
# providers.tf key already updated in code changes
terraform -chdir="$IAC/global/cloud/aws/backend" init -migrate-state -force-copy -input=false
terraform -chdir="$IAC/global/cloud/aws/backend" plan -input=false
echo "[1/4] backend: OK"

#----------------------------------------------------------------------
# Step 3: Migrate global/ecr
#----------------------------------------------------------------------
echo "[2/4] Migrating ecr..."
rm -rf "$IAC/global/aws/ecr/.terraform"
git mv "$IAC/global/aws/ecr" "$IAC/global/cloud/aws/ecr"
terraform -chdir="$IAC/global/cloud/aws/ecr" init -migrate-state -force-copy -input=false
terraform -chdir="$IAC/global/cloud/aws/ecr" plan -input=false
echo "[2/4] ecr: OK"

#----------------------------------------------------------------------
# Step 4: Migrate global/oidc (MUST be before shared)
#----------------------------------------------------------------------
echo "[3/4] Migrating oidc..."
rm -rf "$IAC/global/aws/oidc/.terraform"
git mv "$IAC/global/aws/oidc" "$IAC/global/cloud/aws/oidc"
terraform -chdir="$IAC/global/cloud/aws/oidc" init -migrate-state -force-copy -input=false
terraform -chdir="$IAC/global/cloud/aws/oidc" plan -input=false
echo "[3/4] oidc: OK"

#----------------------------------------------------------------------
# Step 5: Migrate staging/shared (references oidc remote state)
#----------------------------------------------------------------------
echo "[4/4] Migrating shared..."
rm -rf "$IAC/staging/aws/shared/.terraform"
git mv "$IAC/staging/aws/shared" "$IAC/aws/staging/shared"
# data.tf remote_state key already updated in code changes
terraform -chdir="$IAC/aws/staging/shared" init -migrate-state -force-copy -input=false
terraform -chdir="$IAC/aws/staging/shared" plan -input=false
echo "[4/4] shared: OK"

#----------------------------------------------------------------------
# Cleanup empty directories
#----------------------------------------------------------------------
rmdir "$IAC/global/aws" 2>/dev/null || true
rmdir "$IAC/staging/aws" 2>/dev/null || true
rmdir "$IAC/staging" 2>/dev/null || true

echo ""
echo "All migrations complete. Run 'git status' to review changes."
```

## Open Questions

1. **Should old S3 keys be cleaned up after migration?**
   - What we know: `terraform init -migrate-state` copies but does not delete old keys. The S3 bucket has versioning enabled.
   - What's unclear: Whether leaving orphaned keys causes confusion or has cost implications (minimal for small state files).
   - Recommendation: Leave old keys in place as backup. Optionally add a cleanup step at the end of the script that lists old keys and prompts for deletion. Not critical -- state files are a few KB each.

2. **Should state be backed up before migration?**
   - What we know: S3 versioning is enabled on the state bucket. `terraform init -migrate-state` copies (not moves) state. Old keys remain.
   - What's unclear: Whether an explicit `terraform state pull > backup.tfstate` step adds meaningful safety.
   - Recommendation: Skip explicit backup. S3 versioning + the copy-not-move behavior of `-migrate-state` provides sufficient safety. The old S3 keys ARE the backup.

3. **Script execution model: does the planner code the changes then user runs the script, or does the script do everything?**
   - What we know: CONTEXT.md says "migration script handles all 4 moves sequentially". The script is a user-executed manual step.
   - Recommendation: The plan should produce TWO deliverables: (1) Code changes to `.tf` files (new backend keys, new remote state references), committed to git. (2) A migration script that does git mv, deletes .terraform/, runs init -migrate-state, and validates. The user runs the script after the code changes are committed. The script should detect that providers.tf already has the new key and proceed with the migration.

4. **Empty directory cleanup after git mv**
   - What we know: After moving `iac/global/aws/backend`, `iac/global/aws/ecr`, `iac/global/aws/oidc`, the `iac/global/aws/` directory will be empty. After moving `iac/staging/aws/shared/`, `iac/staging/aws/` and `iac/staging/` will be empty. Git does not track empty directories.
   - Recommendation: Script should `rmdir` empty directories after moves. Git will handle this automatically in the commit.
