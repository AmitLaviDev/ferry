**Three composite action.yml files per user's three-action architecture decision:**

1. **`action/setup/action.yml`** — Composite action for dispatch payload parsing:
   - `name: "Ferry Setup"`, `description: "Parse Ferry dispatch payload and output matrix for fan-out"`
   - Inputs: `payload` (required, string — the raw dispatch JSON)
   - Outputs: `matrix` (JSON string for `fromJson()` in strategy)
   - Steps:
     a. `uses: astral-sh/setup-uv@v4` (install uv)
     b. `shell: bash` run: `uv pip install --system ${{ github.action_path }}/..` (install ferry-action package which includes ferry-utils from workspace — note action_path points to setup/ so `..` is the action root)
     c. `shell: bash` run: `python -m ferry_action.parse_payload` with env `INPUT_PAYLOAD: ${{ inputs.payload }}` and id `parse`
   - The setup action output captures matrix from the parse step: `value: ${{ steps.parse.outputs.matrix }}`

2. **`action/build/action.yml`** — Composite action for Docker build + ECR push:
   - `name: "Ferry Build"`, `description: "Build Lambda container and push to ECR"`
   - Inputs: `resource-name` (required), `source-dir` (required), `ecr-repo` (required), `aws-role-arn` (required), `aws-region` (default: "us-east-1"), `trigger-sha` (required), `deployment-tag` (required), `runtime` (default: "python3.12"), `github-token` (default: "", for private repo deps)
   - Outputs: `image-uri` (full ECR URI with tag), `image-digest` (sha256 digest)
   - Steps:
     a. `uses: aws-actions/configure-aws-credentials@v4` with `role-to-assume: ${{ inputs.aws-role-arn }}`, `aws-region: ${{ inputs.aws-region }}`
     b. `shell: bash` run: `python -m ferry_action.build` with env vars for all inputs (INPUT_RESOURCE_NAME, INPUT_SOURCE_DIR, INPUT_ECR_REPO, INPUT_TRIGGER_SHA, INPUT_DEPLOYMENT_TAG, INPUT_RUNTIME, INPUT_GITHUB_TOKEN) and id `build`
   - Outputs capture from build step: image-uri and image-digest

3. **`action/deploy/action.yml`** — Composite action for Lambda deployment:
   - `name: "Ferry Deploy"`, `description: "Deploy Lambda function from ECR image"`
   - Inputs: `resource-name` (required), `function-name` (required), `image-uri` (required), `image-digest` (required), `deployment-tag` (required), `aws-role-arn` (required), `aws-region` (default: "us-east-1")
   - Outputs: `skipped` (true/false), `lambda-version` (published version number)
   - Steps:
     a. `uses: aws-actions/configure-aws-credentials@v4` with `role-to-assume: ${{ inputs.aws-role-arn }}`, `aws-region: ${{ inputs.aws-region }}`
     b. `shell: bash` run: `python -m ferry_action.deploy` with env vars (INPUT_RESOURCE_NAME, INPUT_FUNCTION_NAME, INPUT_IMAGE_URI, INPUT_IMAGE_DIGEST, INPUT_DEPLOYMENT_TAG) and id `deploy`
   - Outputs capture from deploy step: skipped, lambda-version

**NOTE on OIDC auth:** The build and deploy actions each call `aws-actions/configure-aws-credentials@v4` as their first step. This handles the OIDC token exchange — no custom exchange code. The user's workflow must set `permissions: id-token: write`. This satisfies AUTH-02.

**NOTE on uv install:** The setup action installs the ferry-action package via uv. For build and deploy, the ferry-action package was already installed by the setup job (same workflow). But since build/deploy run in separate matrix jobs, they need their own install step. Add `uses: astral-sh/setup-uv@v4` and `uv pip install --system ${{ github.action_path }}/..` as steps before the Python invocation in build and deploy action.yml too.

4. **`action/src/ferry_action/parse_payload.py`** — Dispatch payload parser:
   - Read `INPUT_PAYLOAD` env var
   - Parse with `DispatchPayload.model_validate_json(payload_str)`
   - Filter to only lambda resources (this phase is Lambda-only)
   - Build matrix JSON: `{"include": [{"name": r.name, "source": r.source, "ecr": r.ecr, "trigger_sha": payload.trigger_sha, "deployment_tag": payload.deployment_tag, "runtime": "python3.12"} for r in payload.resources]}`
   - Note: `runtime` is not in the dispatch payload (it's in ferry.yaml which the App parsed). For v1 the runtime is a reasonable default; the build action also accepts it as input so it can be overridden. Use "python3.12" as the default here.
   - Write matrix JSON to `$GITHUB_OUTPUT` using `gha.set_output("matrix", json_str)`
   - Run as `__main__`: `if __name__ == "__main__": main()`
   - Also support `python -m ferry_action.parse_payload` via `__main__` block

5. **`action/src/ferry_action/gha.py`** — GHA logging/output helpers:
   - `set_output(name: str, value: str)` — Append `{name}={value}` to `$GITHUB_OUTPUT` file (handle multiline with delimiter pattern if needed, but matrix JSON is single-line so simple approach fine)
   - `begin_group(title: str)` — Print `::group::{title}`
   - `end_group()` — Print `::endgroup::`
   - `mask_value(value: str)` — Print `::add-mask::{value}`
   - `error(message: str)` — Print `::error::{message}`
   - `warning(message: str)` — Print `::warning::{message}`
   - `write_summary(markdown: str)` — Append markdown to `$GITHUB_STEP_SUMMARY`
   - All functions are thin wrappers around GHA workflow command syntax. Keep them simple, no classes.

6. **Update `action/pyproject.toml`** — Add `boto3` to dependencies (needed by build/deploy scripts in Plans 02/03). Keep `ferry-utils` workspace dep.

7. **Update `action/src/ferry_action/__init__.py`** — Keep minimal, just version.

**IMPORTANT — the `runtime` field gap:** The DispatchPayload/LambdaResource in ferry-utils does NOT include a `runtime` field (it only has name, source, ecr). The ferry.yaml LambdaConfig has `runtime` but it's not propagated to the dispatch payload. For this plan, the setup action outputs a default runtime ("python3.12") in the matrix. The build action accepts `runtime` as an input. This is the correct v1 approach — the dispatch payload is intentionally lean, and runtime is a build concern. Document this in parse_payload.py with a comment.
