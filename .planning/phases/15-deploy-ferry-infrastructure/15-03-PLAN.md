<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Verify Lambda is live and test webhook delivery</name>
  <action>
**Step 1 -- Curl the Function URL:**
```bash
FUNCTION_URL=$(terraform -chdir=iac/aws/staging/us-east-1/ferry_backend output -raw lambda_function_url)
curl -s "$FUNCTION_URL" | jq .
```
**Expected:** A JSON response. Even an error (e.g., missing signature header) proves the Lambda is running the real Ferry backend code, not the placeholder. If you see `{"message": "Ferry backend"}` or similar placeholder response, the self-deploy has not yet run -- proceed to Task 2 first and come back.

**Step 2 -- Test webhook delivery from GitHub App settings:**
1. Go to the GitHub App settings page > **Advanced** tab > **Recent Deliveries**
2. If there are recent deliveries, click one and check the response
3. If no deliveries exist, go back to the **General** tab and find "Webhook" section, click **Redeliver** on a ping event, or push a trivial commit to the ferry repo to generate one

**Expected:** The delivery shows an HTTP response from the Lambda. A 200 is ideal; a 401 (signature mismatch on redeliver) is also fine -- it proves the Lambda received the request and attempted validation.

**Step 3 -- Check CloudWatch logs:**
```bash
aws logs tail /aws/lambda/ferry-backend --since 10m --format short
```
**Expected:** Structured JSON log entries showing the Lambda received and processed requests. Look for entries with `event` or `webhook` or `signature` in the log output.

**NOTE:** If the Lambda is still running the placeholder image (curl returns the placeholder response), this is expected -- the self-deploy in Task 2 will push the real code. After self-deploy completes, re-run these verification steps.
  </action>
  <resume-signal>Confirm: (1) curl returns a response from the Lambda, (2) webhook delivery shows an HTTP response, (3) CloudWatch logs show processing. Describe what you see -- especially whether it is the placeholder or real Ferry code.</resume-signal>
</task>

<task type="checkpoint:human-action" gate="blocking">
  <name>Task 2: Trigger and verify the self-deploy pipeline</name>
  <action>
**Step 1 -- Trigger the self-deploy workflow by pushing a trivial commit:**

Push any commit to main. A simple option:
```bash
git commit --allow-empty -m "chore: trigger self-deploy pipeline"
git push origin main
```

**Step 2 -- Monitor the workflow run:**
Go to: `https://github.com/<owner>/ferry/actions/workflows/self-deploy.yml`

Watch both jobs:
1. **test** job: Should install dependencies and run pytest successfully
2. **deploy** job: Should configure AWS credentials via OIDC, login to ECR, build Docker image, push to ECR, and update Lambda function code

**Step 3 -- If the workflow fails, diagnose:**
- **"Configure AWS credentials" step fails:** Check that `AWS_DEPLOY_ROLE_ARN` repo secret is correct. Verify the OIDC trust policy `sub` claim matches the ferry repo name (case-sensitive).
- **"Login to Amazon ECR" step fails:** Check that the deploy role has `ecr:GetAuthorizationToken` permission.
- **"Build and push" step fails:** Check Dockerfile build errors in the logs.
- **"Update Lambda function code" step fails:** Check that the deploy role has `lambda:UpdateFunctionCode` permission and the function name is correct.

**Step 4 -- After successful deploy, re-verify Lambda is serving real code:**
```bash
FUNCTION_URL=$(terraform -chdir=iac/aws/staging/us-east-1/ferry_backend output -raw lambda_function_url)
curl -s "$FUNCTION_URL" | jq .
```
The response should now be from the real Ferry backend (likely an error about missing/invalid webhook signature, which is correct behavior).

**Step 5 -- Re-test webhook delivery:**
1. Go to GitHub App settings > Advanced > Recent Deliveries
2. Click **Redeliver** on any event
3. Check the response code and body

**Expected:** An HTTP response from the real Ferry code. A 401 with signature validation error is normal for redelivered events.

**Step 6 -- Verify CloudWatch logs show real Ferry processing:**
```bash
aws logs tail /aws/lambda/ferry-backend --since 5m --format short
```
Look for structured log entries from the Ferry backend (not placeholder).
