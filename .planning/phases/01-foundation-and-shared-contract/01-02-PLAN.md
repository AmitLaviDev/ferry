<feature>
  <name>Webhook Signature Validation</name>
  <files>backend/src/ferry_backend/webhook/signature.py, tests/test_backend/test_signature.py</files>
  <behavior>
    verify_signature(body: str, signature_header: str, secret: str) -> bool

    Cases:
    - Valid HMAC-SHA256 signature -> True
    - Tampered body (different body, same signature) -> False
    - Missing signature header (empty string) -> False
    - Signature without "sha256=" prefix -> False
    - Wrong secret -> False

    Implementation details:
    - Use hmac.new() with hashlib.sha256 (stdlib) per RESEARCH.md Pattern 1
    - Use hmac.compare_digest() for constant-time comparison (prevent timing attacks)
    - Signature format: "sha256=" + hex digest
    - Encode both secret and body as utf-8 before HMAC computation
  </behavior>
  <implementation>
    Minimal function: compute expected HMAC, compare with provided signature using constant-time comparison.
    Handle edge cases: missing prefix, empty signature, encoding.
    See RESEARCH.md Pattern 1 for reference implementation.
  </implementation>
</feature>

<feature>
  <name>DynamoDB Dual-Key Deduplication</name>
  <files>backend/src/ferry_backend/webhook/dedup.py, tests/test_backend/test_dedup.py</files>
  <behavior>
    is_duplicate(delivery_id: str, payload: dict, table_name: str, client: DynamoDB client) -> bool

    Cases:
    - First delivery with unique delivery_id -> False (not duplicate, record created)
    - Same delivery_id again -> True (duplicate, ConditionalCheckFailedException caught)
    - New delivery_id but same repo+SHA (re-queued event) -> True (event-level dedup catches it)
    - First delivery with missing repo/SHA in payload -> False (delivery-level dedup only, skip event-level)
    - TTL field set to 24 hours from now on all records

    Implementation details:
    - Dual-key dedup: delivery-level (DELIVERY#{delivery_id}) AND event-level (EVENT#push#{repo}#{after_sha})
    - DynamoDB conditional write: ConditionExpression='attribute_not_exists(pk)'
    - ConditionalCheckFailedException is the expected signal for duplicate — catch explicitly, not as error
    - TTL_SECONDS = 86400 (24 hours per user decision)
    - pk/sk pattern: pk = dedup key, sk = "METADATA"
    - Records include: pk, sk, expires_at (TTL), created_at
    - Accept boto3 DynamoDB client as parameter (for testability with moto)
  </behavior>
  <implementation>
    Two-step dedup: try delivery-level write first (most common), then event-level write.
    _try_record helper does the conditional put_item and catches ConditionalCheckFailedException.
    _build_event_key extracts repo full_name and after SHA from push event payload.
    See RESEARCH.md Pattern 2 for reference implementation.
  </implementation>
</feature>

<feature>
  <name>Lambda Webhook Handler</name>
  <files>backend/src/ferry_backend/webhook/handler.py, tests/test_backend/test_handler.py</files>
  <behavior>
    handler(event: dict, context) -> dict

    Cases:
    - Valid signature + new delivery -> 200 {"status": "accepted"}
    - Valid signature + duplicate delivery -> 200 {"status": "duplicate"}
    - Invalid/missing signature -> 401 {"error": "invalid signature"}
    - Non-push event type -> 200 {"status": "ignored"} (Phase 1: push only)
    - Base64-encoded body (isBase64Encoded: true) -> decoded before signature validation
    - Missing x-github-delivery header -> 400 {"error": "missing delivery id"}

    Implementation details:
    - Lambda Function URL event format (payload v2): body in event['body'], headers in event['headers']
    - Normalize headers to lowercase immediately (RESEARCH.md Pitfall 5)
    - Handle isBase64Encoded flag (RESEARCH.md Pitfall 1)
    - Validate signature BEFORE any JSON parsing (RESEARCH.md anti-pattern)
    - Parse payload with json.loads after validation
    - Check event type from x-github-event header — only process "push" in Phase 1
    - Settings loaded at module level (Lambda cold start optimization)
    - Log with structlog: delivery_id, event_type, repo bound to context
    - DynamoDB client created at module level for connection reuse across invocations
  </behavior>
  <implementation>
    Wire together signature.verify_signature, dedup.is_duplicate, and event parsing.
    Module-level Settings() and boto3 client for Lambda cold start optimization.
    Structured logging with request context.
    Return Lambda Function URL response format: {statusCode, headers, body}.
    Phase 1 stub: after dedup, just return "accepted". Phase 2 adds config reading, change detection, dispatch.
  </implementation>
</feature>
