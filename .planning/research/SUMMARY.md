# Project Research Summary

**Project:** Ferry v1.1 — Deploy to AWS Staging
**Domain:** Terraform IaC for deploying a GitHub App Lambda backend to AWS
**Researched:** 2026-02-28
**Confidence:** MEDIUM-HIGH

## Executive Summary

Ferry v1.1 is a focused infrastructure task: provision the AWS resources needed to run the existing Ferry backend Lambda in a staging environment, then wire up automated self-deployment via GitHub Actions. The approach is deliberately thin — four Terraform projects arranged by dependency (bootstrap, ECR, shared IAM, Lambda backend), with a self-deploy workflow that builds the container, pushes to ECR, and updates the Lambda on every push to main. Raw Terraform resources are preferred over community modules for this scale (5-6 AWS resources total); the `terraform-aws-modules` abstractions add cognitive overhead with no payoff at this resource count.

The key risk is operational sequencing: there are three interdependent bootstrap problems that must be resolved in strict order. Terraform needs an S3 bucket before it can store state. The Lambda needs a container image before it can be created. The GitHub App needs a webhook URL before it can be registered — but the URL is generated by Terraform. Each of these is a one-time manual coordination step, not an ongoing concern. Documenting and following the nine-step setup sequence from the PITFALLS research is the primary mitigation.

The recommended secrets strategy is to have Terraform create Secrets Manager secret containers while secret values are populated manually via CLI — never in Terraform state. Lambda reads secrets via environment variables sourced from Terraform data sources at cold start (simpler than the Secrets Manager Lambda Extension and sufficient for staging). One code change is required: `settings.py` must accept Secrets Manager ARNs as environment variables rather than direct secret values.

## Key Findings

### Recommended Stack

Ferry v1.1 uses Terraform >= 1.9.0 with the hashicorp/aws provider ~> 5.80, targeting us-east-1. Raw AWS resources are preferred over terraform-aws-modules for all components — the lambda, dynamodb-table, and ecr community modules are designed for teams managing dozens of resources, and Ferry's infra (one Lambda, one DynamoDB table, one Function URL, one ECR repo) does not justify the abstraction overhead. The ConvergeBio/iac-tf convention of raw resource definitions with consistent file splits (providers.tf, main.tf, variables.tf, outputs.tf, data.tf, locals.tf) applies throughout. The Lambda runs on arm64 (Graviton) for a 20% cost reduction.

**Core technologies:**
- Terraform >= 1.9.0: IaC engine — current stable branch with improved input validation
- hashicorp/aws ~> 5.80: AWS resource management — mature 5.x line with Lambda Function URL support since 4.x
- S3 + DynamoDB: Terraform state backend + locking — self-hosted, free, proven; no external service dependency
- Lambda container image (arm64/Graviton): Ferry backend compute — 20% cost reduction, faster Python cold starts
- Lambda Function URL (auth=NONE): Webhook endpoint — free, zero additional resources vs API Gateway, HMAC auth handled in app code
- Secrets Manager: GitHub App credentials — JSON-structured secrets, supports rotation, $0.40/month per secret

**Version verification needed at implementation time:** AWS provider exact version (may be beyond 5.80 by 2026-02-28), Python 3.14 arm64 Lambda base image availability on public.ecr.aws.

### Expected Features

Research derived feature requirements directly from the existing Ferry backend source code (settings.py, handler.py, dedup.py). These are not speculative — the app code establishes hard contracts the IaC must satisfy.

**Must have (table stakes):**
- S3 State Backend + DynamoDB Lock — Terraform CI/CD requires remote state; lock table prevents concurrent apply corruption
- ECR Repository (`ferry/backend`) with lifecycle policy — container registry; lifecycle policy (keep last 10 images) controls storage costs
- Lambda Function (container image, arm64, 256MB, 30s timeout) — webhook handler compute resource
- Lambda Function URL (auth=NONE) — GitHub webhook endpoint; IAM auth would block GitHub delivery
- DynamoDB Table for dedup (pk/sk, TTL on expires_at, PAY_PER_REQUEST) — app code expects this schema; TTL prevents unbounded growth
- Secrets Manager secret containers for GitHub App credentials — values populated manually, never in Terraform state
- IAM Execution Role with least-privilege policies — DynamoDB PutItem/GetItem, secretsmanager:GetSecretValue (scoped to specific ARNs), CloudWatch Logs
- OIDC Identity Provider + GHA Deploy Role — keyless AWS auth for self-deploy workflow; scoped to main branch
- Backend Dockerfile (`backend/Dockerfile.deploy`) — new file needed; distinct from the Magic Dockerfile; builds from repo root context to access both utils/ and backend/ packages
- Self-Deploy GHA Workflow (`.github/workflows/deploy-ferry.yml`) — OIDC auth + ECR login + docker build+push + `aws lambda update-function-code` on push to main

**Should have (operational quality):**
- CloudWatch Alarms (5xx rate, p99 latency) — catch issues before GitHub starts retrying at scale
- X-Ray active tracing — trace cold starts and GitHub API call latency
- Lambda reserved concurrency (10-25) — blast radius control; returns 429 to GitHub (GitHub retries automatically)
- CloudWatch Dashboard — single-pane operational view: invocations, errors, duration, DynamoDB consumed capacity
- Resource tagging via provider default_tags — Project=ferry, Environment=staging, ManagedBy=terraform
- Terraform variable separation (tfvars) — staging.tfvars enables future prod environment without duplicating TF code

**Defer (v2+ only):**
- API Gateway in front of Function URL — unnecessary until custom domain, WAF, or multi-endpoint needs arise
- VPC configuration — Lambda accesses all services via public endpoints; VPC adds 1-2s cold start and $32+/month for NAT Gateway
- Secrets Manager Lambda Extension — overkill for staging; direct env var approach from Terraform data sources is sufficient
- Lambda provisioned concurrency — accept cold starts; GitHub tolerates 10s; revisit for production
- Custom domain (CloudFront + ACM + Route53) — auto-generated Function URL is fine for staging
- DLQ — Function URL invocations are synchronous; DLQ only applies to async invocations
- KMS Customer Managed Keys — default AWS encryption sufficient for staging

### Architecture Approach

The deployment is structured as four Terraform projects with a strict dependency chain: bootstrap (S3 bucket + DynamoDB lock table) → ECR → shared (IAM roles, Secrets Manager, OIDC provider) → ferry_backend (Lambda + Function URL + DynamoDB dedup table). Projects reference each other via `terraform_remote_state` data sources — the standard ConvergeBio/iac-tf pattern. Two new files live outside the IaC directory: `backend/Dockerfile.deploy` (containerizes the ferry-backend Python package) and `.github/workflows/deploy-ferry.yml` (the self-deploy pipeline). One code change is required: `settings.py` must accept Secrets Manager ARNs as environment variables and load actual values at cold start.

**Major components:**
1. Bootstrap TF project (`iac/global/cloud/aws/backend/`) — S3 state bucket + DynamoDB lock table; applied once with local state then migrated to itself via `terraform init -migrate-state`
2. ECR TF project (`iac/global/cloud/aws/ecr/`) — container image registry; placed in global/ (not per-environment) because the same images serve staging and production
3. Shared TF project (`iac/teams/platform/aws/staging/shared/`) — Lambda IAM execution role, OIDC provider + GHA deploy role, Secrets Manager secret containers (values populated manually)
4. Ferry Backend TF project (`iac/teams/platform/aws/staging/us_east_1/ferry_backend/`) — Lambda function, Function URL, DynamoDB dedup table, CloudWatch log group; reads upstream state via `terraform_remote_state`
5. Self-Deploy GHA Workflow — OIDC auth → ECR login → docker build+push → `aws lambda update-function-code`; owns code lifecycle after Terraform creates the Lambda shell
6. Backend Dockerfile (`backend/Dockerfile.deploy`) — `FROM public.ecr.aws/lambda/python:3.14`, installs ferry-utils then ferry-backend packages from repo root context

**Key patterns:**
- `lifecycle { ignore_changes = [image_uri] }` on Lambda — Terraform owns infra configuration; GHA owns deployed code; without this, every `terraform apply` reverts the Lambda to whatever image is in state
- `terraform_remote_state` for cross-project references — state key mirrors directory path; outputs treated as a versioned contract
- Environment-scoped resource naming (`ferry-backend-staging`) — enables prod environment without naming conflicts
- Secrets Manager containers in Terraform, values outside Terraform — secrets never appear in state files or version control

### Critical Pitfalls

1. **S3 backend bootstrap (chicken-and-egg)** — Apply the bootstrap project with local state first, create the S3 bucket, then run `terraform init -migrate-state` to move local state into S3. This is a one-time manual sequence. Never delete the bootstrap bucket; enable versioning from day one.

2. **Self-deploy circular dependency** — Lambda with `PackageType: Image` requires a valid image URI at creation time. Pull the public Lambda Python base image, tag it, and push it to ECR as a placeholder before running `terraform apply` on ferry_backend. Add `lifecycle { ignore_changes = [image_uri] }` so the self-deploy workflow's subsequent updates do not get reverted.

3. **OIDC trust policy sub claim mismatch** — The `sub` claim from GitHub Actions is `repo:{owner}/{repo}:ref:refs/heads/{branch}` for branch-triggered workflows. Getting owner case, format, or condition operator wrong causes silent `AssumeRoleWithWebIdentity` failures. Start with a permissive `StringLike` condition (`repo:your-org/ferry:*`), verify with `aws sts get-caller-identity` as the first workflow step, then tighten to the specific branch.

4. **Secrets Manager lifecycle mismatch** — Terraform creates the secret container resource; actual secret values (GitHub App private key, webhook secret, app ID, installation ID) are populated manually via `aws secretsmanager put-secret-value`. Never put secret values in Terraform variables or `aws_secretsmanager_secret_version` resources — they end up in state files in plaintext.

5. **GitHub App registration timing** — The Function URL is auto-generated by AWS and unknown until after `terraform apply`. The safe sequence: deploy all infra (including the Lambda) → note the Function URL output → register the GitHub App using that URL → populate Secrets Manager values → trigger the first real self-deploy. Alternatively, register with a placeholder URL first and update it afterward.

6. **Lambda IAM role missing permissions** — IAM creation succeeds even with an incomplete policy; failures surface at runtime as `AccessDeniedException` in CloudWatch logs. Define the complete policy upfront: DynamoDB (PutItem, GetItem scoped to dedup table ARN), secretsmanager:GetSecretValue (scoped to specific secret ARNs), CloudWatch Logs. Verify with a real smoke invocation — moto does not validate IAM.

7. **DynamoDB billing mode default** — Terraform's `aws_dynamodb_table` defaults to `billing_mode = "PROVISIONED"` with zero capacity, creating a table that throttles immediately. Always set `billing_mode = "PAY_PER_REQUEST"` explicitly.

## Implications for Roadmap

### Phase 1: Bootstrap + Global Resources

**Rationale:** The S3 state backend and ECR repository are prerequisites for every subsequent Terraform project. The bootstrap problem (chicken-and-egg with S3 state) requires a two-step local→S3 migration that must be completed before anything else can be initialized. ECR must exist before a placeholder image can be pushed, and a placeholder image must exist before the Lambda can be created. These two projects have no dependencies on each other beyond the S3 backend, so the TF code can be written in parallel but must be applied sequentially.

**Delivers:** Remote Terraform state with locking (S3 + DynamoDB lock table), container image registry with lifecycle policy (`ferry/backend`), and a pushed placeholder image that unblocks Lambda creation.

**Addresses:** S3 State Backend, DynamoDB Lock Table, ECR Repository (all table stakes from FEATURES.md)

**Avoids:** Pitfalls 1, 2, 7, 9, 14 — bootstrap chicken-and-egg, lock table confusion, image circular dependency, no lifecycle policy on ECR, state key path conflicts

**Research flag:** Standard well-documented patterns. Skip research-phase.

### Phase 2: Shared IAM + Secrets

**Rationale:** The Lambda execution role and OIDC provider must exist before the Lambda and GHA workflow can function. Secrets Manager containers must be created before the Lambda references their ARNs as environment variables. This phase has the highest concentration of correctness-sensitive IAM patterns: OIDC thumbprint field (required but not validated by AWS), sub claim format (case-sensitive, format-sensitive, silent failures), and least-privilege policy completeness (fails at runtime not deploy time).

**Delivers:** Lambda IAM execution role with correct permissions (DynamoDB, Secrets Manager, CloudWatch Logs), OIDC provider + GHA deploy role (ECR push + Lambda update), Secrets Manager secret containers (values populated manually after GitHub App registration), resource tagging via default_tags.

**Addresses:** IAM Execution Role, OIDC Identity Provider + GHA Deploy Role, Secrets Manager Secrets (table stakes); resource tagging (differentiator)

**Avoids:** Pitfalls 3, 4, 6, 10 — OIDC thumbprint confusion, sub claim mismatch (start permissive then tighten), secrets lifecycle mismatch (containers only, no values in TF), missing Lambda IAM permissions

**Research flag:** Standard IAM patterns. The OIDC sub claim format is the only nuanced area and is fully covered in PITFALLS.md. Skip research-phase.

### Phase 3: Ferry Backend Core (Lambda + DynamoDB + Function URL)

**Rationale:** This is the primary deployment target. All dependencies from Phases 1 and 2 must be applied before this phase can be planned or applied. The Lambda references the ECR repo URL (from Phase 1 remote state), the IAM role ARN (from Phase 2 remote state), and the Secrets Manager ARNs (from Phase 2 remote state). The DynamoDB dedup table and CloudWatch log group are created alongside the Lambda in this project.

**Delivers:** Running Lambda function accessible via a known Function URL, DynamoDB dedup table with TTL, structured logging via CloudWatch (30-day retention for staging), Lambda configuration matching all contracts from existing app code.

**Addresses:** Lambda Function, Lambda Function URL, DynamoDB Table + TTL, CloudWatch Log Group (all table stakes from FEATURES.md)

**Avoids:** Pitfalls 5, 8, 13, 15, 16, 17 — Function URL auth exposure (reserved concurrency + monitoring), payload format mismatch (explicitly document v2.0 format), hardcoded ARNs in env vars (use TF resource references), DynamoDB PAY_PER_REQUEST (explicit, not default), 30s timeout + 256MB memory, Function URL output captured for GitHub App registration

**Research flag:** Standard patterns with complete HCL snippets in STACK.md and ARCHITECTURE.md. Skip research-phase.

### Phase 4: Self-Deploy Pipeline

**Rationale:** The deployment automation (Backend Dockerfile + GHA workflow) can only be finalized after the Lambda and ECR repo exist, because the GHA workflow needs the OIDC role ARN and the ECR repo URL from Terraform outputs. This phase also encompasses the one-time manual setup sequence: populate Secrets Manager values, register the GitHub App using the Function URL output, trigger the first real self-deploy. The Backend Dockerfile must be built from the repo root (not from `backend/`) because it needs both utils/ and backend/ packages.

**Delivers:** `backend/Dockerfile.deploy` (FROM public Lambda Python base, installs ferry-utils + ferry-backend), `.github/workflows/deploy-ferry.yml` (OIDC → ECR login → docker build+push → `aws lambda update-function-code`), one-time GitHub App registration runbook, first verified end-to-end deploy.

**Addresses:** Backend Dockerfile, Self-Deploy GHA Workflow (table stakes); tfvars separation for future prod (differentiator)

**Avoids:** Pitfalls 11, 18, 19 — GitHub App registration timing (accept sequential deploy-first then register), missing `id-token: write` permission in GHA workflow (explicit permissions block is mandatory), ECR region mismatch (consistent region variable throughout)

**Research flag:** Standard GHA + ECR + Lambda patterns. The Backend Dockerfile is simpler than the Magic Dockerfile (no build secrets, no system packages). Skip research-phase.

### Phase 5: Operational Readiness

**Rationale:** Once the Lambda is receiving real webhook traffic, operational visibility pays off quickly. CloudWatch alarms, X-Ray tracing, and reserved concurrency are all low-effort adds that improve mean time to detection and limit blast radius from unexpected traffic. These are additive changes that carry no deployment risk — they can be applied independently without touching the core Lambda or DynamoDB configuration.

**Delivers:** CloudWatch alarms (5xx rate > 0, p99 latency > 5s) with SNS notification, X-Ray active tracing (mode=Active, AWSXRayDaemonWriteAccess policy attachment), reserved concurrency set to 10-25, CloudWatch dashboard (invocations, errors, duration, DynamoDB consumed capacity).

**Addresses:** All differentiators from FEATURES.md (alarms, X-Ray, reserved concurrency, dashboard)

**Avoids:** Pitfall 5 secondary mitigation — reserved concurrency limits blast radius if the Function URL receives unexpected traffic

**Research flag:** Standard AWS observability patterns with clear Terraform resource mappings. Skip research-phase.

### Phase Ordering Rationale

- Phases 1 and 2 must precede Phase 3: hard dependency chain (state backend → ECR → placeholder image → Lambda; IAM role → Lambda; Secrets → Lambda env vars). Code can be written in parallel but `terraform apply` order is strict.
- Phase 4 requires a running Lambda endpoint: the GHA workflow needs the deployed OIDC role ARN and ECR repo URL from Terraform outputs; the GitHub App registration requires the actual Function URL.
- Phase 5 is purely additive: can be applied any time after Phase 3 is deployed and receiving traffic; carries no risk to existing resources.
- The nine-step manual setup sequence from PITFALLS.md spans Phases 1-4 and should be written as a runbook committed to the repo alongside the Terraform code.

### Research Flags

Phases needing deeper research during planning: **None** — all five phases use established, well-documented patterns. The research files contain sufficient implementation detail, including complete HCL resource blocks, a GHA workflow file, and a Dockerfile skeleton.

Phases with standard patterns (skip research-phase): **All five phases**. Terraform S3 backend bootstrap, Lambda container image deployment with `ignore_changes`, OIDC federation for GHA, ECR lifecycle policies, and self-deploy workflows are mature patterns with extensive official documentation and real-world precedent in the ConvergeBio/iac-tf and pipelines-hub reference implementations.

### Gaps to Address

- **Python 3.14 Lambda base image availability:** `public.ecr.aws/lambda/python:3.14` may not exist for arm64 as of implementation. Fallback: use Python 3.13 (likely available) or build on `public.ecr.aws/lambda/provided:al2023` with Python 3.14 installed. Verify before writing the Dockerfile.

- **AWS provider version:** Training data covers the ~5.80 range. Verify `registry.terraform.io/providers/hashicorp/aws/latest` at implementation time. Use the `~>` pessimistic constraint operator so patch updates are picked up automatically without a manual bump.

- **settings.py Secrets Manager integration:** The research identified that `FERRY_PRIVATE_KEY` and `FERRY_WEBHOOK_SECRET` should be loaded via Secrets Manager ARN env vars rather than direct value env vars. This requires a code change to ferry-backend, not just IaC. Confirm the current settings.py interface before finalizing the Lambda environment variable block in Phase 3.

- **OIDC trust policy owner string:** The exact GitHub org/user for `token.actions.githubusercontent.com:sub` is case-sensitive and must match the real owner. All STACK.md and ARCHITECTURE.md snippets use placeholder `OWNER`. Verify with `aws sts get-caller-identity` as the first step in the first self-deploy run.

- **ARCHITECTURE.md module vs raw resource discrepancy:** ARCHITECTURE.md references `terraform-aws-modules/lambda/aws`, `terraform-aws-modules/ecr/aws`, and `terraform-module/github-oidc-provider/aws`. STACK.md explicitly recommends raw resources. **Decision: use raw resources** as recommended by STACK.md — consistent with the small resource count and the ConvergeBio/iac-tf convention of explicit resource definitions. ARCHITECTURE.md module references should be ignored in favor of STACK.md's raw resource examples.
