# Roadmap: Ferry v1.5 Batched Dispatch

## Overview

v1.5 replaces per-type workflow_dispatch (one dispatch per resource type, producing N workflow runs with skipped-job clutter) with a single batched dispatch per push. One push produces one workflow run with only the relevant deploy jobs active. This touches the shared data contract (new payload model), backend dispatch logic (merge types into single payload), action parsing (output per-type matrices and boolean flags), and the workflow template (gate on booleans, use per-type matrices).

## Phases

- [ ] **Phase 25: Shared Models and Schema** - Add BatchedDispatchPayload model and bump schema version to 2
- [ ] **Phase 26: Backend Batched Dispatch** - Replace per-type dispatch loop with single batched dispatch, add payload-size fallback
- [ ] **Phase 27: Action Parsing and Workflow Template** - Parse batched payloads into per-type outputs; update ferry.yml template with boolean gates and per-type matrices
- [ ] **Phase 28: E2E Validation** - Prove batched dispatch works end-to-end with test repo

## Phase Details

### Phase 25: Shared Models and Schema
**Goal**: Both backend and action can import and use the batched dispatch data contract
**Depends on**: Nothing (first phase of v1.5)
**Requirements**: DISP-02
**Success Criteria** (what must be TRUE):
  1. `BatchedDispatchPayload` model exists with typed per-type resource lists (lambdas, step_functions, api_gateways) and `v=2` field
  2. Existing `DispatchPayload` (v1) model is preserved unchanged for backward compatibility
  3. All model tests pass including round-trip serialization of the new batched payload
**Plans**: TBD

### Phase 26: Backend Batched Dispatch
**Goal**: Backend sends one workflow_dispatch per push containing all affected resource types in a single payload
**Depends on**: Phase 25
**Requirements**: DISP-01, DISP-03
**Success Criteria** (what must be TRUE):
  1. A push affecting multiple resource types produces exactly one workflow_dispatch call (not one per type)
  2. The dispatched payload contains all affected resource types grouped by type (lambdas, step_functions, api_gateways lists)
  3. If the combined payload exceeds 65,535 characters, the backend falls back to per-type dispatch (one dispatch per type, v1 behavior)
  4. All backend dispatch and handler tests pass with the new single-dispatch model
**Plans**: TBD

### Phase 27: Action Parsing and Workflow Template
**Goal**: Setup action parses batched payloads and outputs per-type matrices and boolean flags; ferry.yml template uses these outputs to gate deploy jobs
**Depends on**: Phase 25
**Requirements**: ACT-01, ACT-02, ACT-03, ACT-04, TMPL-01, TMPL-02, TMPL-03
**Success Criteria** (what must be TRUE):
  1. Setup action outputs boolean flags (has_lambdas, has_step_functions, has_api_gateways) indicating which types have resources
  2. Setup action outputs separate per-type matrix JSON (lambda_matrix, sf_matrix, ag_matrix) consumable by `fromJson()`
  3. Setup action outputs resource_types comma-separated string for workflow run-name display
  4. Setup action correctly handles both v1 (per-type) and v2 (batched) payloads -- v1 payloads produce legacy outputs, v2 payloads produce batched outputs
  5. Updated ferry.yml template gates deploy jobs on boolean flags and references per-type matrix outputs
**Plans**: TBD

### Phase 28: E2E Validation
**Goal**: Batched dispatch works end-to-end with the test repo -- one push, one workflow run, correct jobs active
**Depends on**: Phase 26, Phase 27
**Requirements**: VAL-01, VAL-02
**Success Criteria** (what must be TRUE):
  1. A push touching all 3 resource types in the test repo produces exactly 1 workflow run with all 3 deploy jobs active (no skipped deploy jobs)
  2. A push touching only 1 resource type produces exactly 1 workflow run with 1 active deploy job and 2 skipped deploy jobs
**Plans**: TBD
